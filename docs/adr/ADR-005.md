# ADR-005: Lightweight RAG Pipeline Assembly with Factory Pattern and Evaluation Foundation

**Status**: Accepted  
**Date**: 2026-06-08  
**Deciders**: Ivan Beira

## Context

Following the implementation of rich hierarchical chunking (ADR-002) and a metadata-aware hybrid retriever using Reciprocal Rank Fusion and configurable boosting (ADR-004), the project has solid, well-tested retrieval building blocks. However, there is still no clean, maintainable way to **assemble** these components into a working, end-to-end RAG system that can be:

- Easily exercised against the full Merck Manual for the four clinical questions.
- Extended with generation, evaluation, and safety features without accumulating technical debt.
- Reproduced consistently across experiments, demos, and future production iterations.

A pragmatic but senior-level approach is required: leverage LangChain’s strengths (LCEL, `create_retrieval_chain`) while keeping the assembly layer thin, explicit, and safe for a clinical domain.

## Decision

We will introduce a **lightweight RAG Pipeline Assembly layer** using LangChain components combined with a **centralized Factory Pattern**.

All factory logic lives in a single file:

- `src/frontline_clinical_rag/pipeline/factory.py`

### Scope of the Factory (What It Does and Does NOT Do)

The factory’s only job is **composition and configuration wiring**. It is deliberately thin.

`create_hybrid_retriever()` (initial function) is responsible for:

- Reading all relevant settings from the single source of truth `core/config.py` (embedding model name/dimensions, vector store backend + persistence path, hybrid weights, metadata boost rules, top-k, chunking strategy metadata, etc.).
- Instantiating the embedding model (OpenAI or local via config).
- Initializing or loading the vector store (Chroma or FAISS) that already contains the hierarchical chunks + rich clinical metadata produced by ADR-002.
- Composing the metadata-aware hybrid retriever (dense + sparse/BM25 + Reciprocal Rank Fusion + configurable field boosting) as specified in ADR-004.
- Returning a ready-to-use retriever object (LangChain `BaseRetriever` or equivalent) that can be passed directly to `create_retrieval_chain` or used standalone for inspection.

The factory **explicitly must not**:

- Implement any retrieval scoring, fusion algorithm, or metadata boosting logic (these belong in the `retrieval/` package).
- Define, store, or render prompt templates / system messages (these will live in `prompts/` or a dedicated `chain/` layer later).
- Contain medical/clinical decision rules, disclaimers, or safety logic.
- Perform heavy I/O or side effects beyond what is strictly required to instantiate configured components.
- Introduce global state or hidden singletons.

### Dependency Boundaries (Architectural Rule)

To keep the system maintainable and testable:

- `pipeline/factory.py` **is allowed to depend on**:
  - `core/config.py`
  - `retrieval/hybrid_retriever.py` (or the module that exposes the hybrid composition)
  - LangChain embedding / vector store abstractions
- `retrieval/`, `chunking/`, `core/`, and `embeddings/` **must never import from `pipeline/`**

This one-way dependency rule prevents circular imports and makes the retrieval layer independently testable and reusable (e.g., for pure retrieval benchmarks).

### Testability

Factory functions will be written to accept optional configuration overrides (e.g., `config: Config | None = None` or a small `overrides: dict`). This enables:

- Easy unit testing with in-memory vector stores and mocked embeddings.
- Fast feedback loops without touching real OpenAI keys or large persisted indexes.
- Clear separation between “how components are wired” and “how components behave”.

No global state or module-level singletons will be used.

### Clinical Safety & Production Readiness (Non-Negotiable for Medical RAG)

This is a **clinical decision-support RAG system**, not a general-purpose chatbot. Therefore:

> Any generation-oriented chains or pipelines created from this factory (future `create_rag_chain()`, `create_generation_step()`, etc.) **must be treated as non-production** until the following are implemented, validated, and documented:
>
> - Comprehensive evaluation harness (RAGAS metrics + clinical-specific measures: citation faithfulness, medical entity grounding, refusal behavior on unsafe/out-of-scope queries, consistency across rephrasings).
> - Mandatory source citation in every generated answer with verifiable page/section references from the Merck Manual.
> - Output guardrails / refusal mechanisms for queries that are ambiguous, contradictory to retrieved context, or request advice outside the knowledge base.
> - Defined human-in-the-loop review process for any high-stakes clinical use.

This explicit disclaimer protects patients, clinicians, regulatory posture, and the long-term credibility of the portfolio project.

### Initial Scope & Implementation Roadmap

**Phase 1 (this ADR)**:
- `pipeline/factory.py` with `create_hybrid_retriever()`
- Thin demonstration script or notebook that runs the four clinical questions end-to-end and prints retrieved chunks + sources (visibility first)

**Phase 2 (future ADR)**:
- Generation layer + prompt management
- Evaluation harness
- Safety guardrails
- Observability hooks

TODO comments may be used for short-term implementation reminders inside the factory, but architectural decisions and safety constraints live in this ADR (and future ones).

## Consequences

**Positive**
- Provides one obvious place to assemble and configure the RAG system.
- Dramatically improves maintainability and reduces duplication compared with scattering wiring logic across notebooks or `run.py`.
- Creates a natural, low-friction extension point for generation, evaluation, and guardrails.
- Enforces clear dependency direction and testability from day one.
- Makes the elevated safety bar for clinical AI explicit and non-negotiable — essential for a serious medical RAG portfolio piece.
- Keeps the architecture pragmatic and aligned with LangChain’s strengths rather than fighting them.

**Negative / Trade-offs**
- Introduces a thin assembly layer (intentionally kept minimal; we rejected heavier abstractions).
- Full end-to-end retrieval + generation will arrive in a controlled follow-up iteration rather than in one big step. This is a deliberate, safety-conscious trade-off.
- Requires a small amount of upfront package organization (`src/frontline_clinical_rag/pipeline/`, `retrieval/`, etc.).

## Alternatives Considered

| Approach                                      | Pros                                      | Cons                                                                 | Decision     |
|-----------------------------------------------|-------------------------------------------|----------------------------------------------------------------------|--------------|
| Everything in notebooks / `run.py`            | Fastest initial implementation            | Becomes unmaintainable; no reproducibility; hard to add eval/safety  | Rejected     |
| Multiple factory files (`retrieval/` + `pipeline/`) | Stronger separation of concerns      | Overkill for current scope; adds cognitive load without benefit      | Rejected     |
| **Centralized lightweight factory in `pipeline/`** | Simple, pragmatic, extensible, clear boundaries | Slightly less “pure” than split factories                       | **Chosen**   |
| Heavy custom pipeline (minimal LangChain)     | Full control over every detail            | Reinvents well-solved problems; slower development; higher maintenance | Rejected     |
| Put generation + eval in the same factory immediately | “Complete” pipeline faster            | Violates the safety disclaimer; risks shipping un-evaluated clinical generation | Rejected     |
