# ADR-007: LangGraph-orchestrated Clinical RAG Pipeline with Safety Critic Node and Evaluation Foundation

**Status:** Completed 
**Date:** 2026-06-10  
**Deciders:** Ivan Beira  
**Consulted:** —  
**Informed:** Portfolio reviewers / future maintainers

## Context
ADR-005 delivered a thin, config-driven factory for hybrid retrieval (hierarchical/recursive strategies, metadata-aware RRF + boosting).  
ADR-006 delivered a dedicated post-generation Safety & Validation layer (`safety/` package, Pydantic `ClinicalResponse` schema, deterministic `SafetyCritic`, input guardrails, `apply_safety_layer` hook) with the explicit principle that generation-oriented flows must remain non-production until evaluation and guardrails exist.

We now have composable, testable retrieval and safety primitives. What is missing for a credible, production-oriented clinical decision-support demonstration is:
- A stateful orchestration layer that reliably chains retrieval → (grounded generation) → safety critic → structured output.
- An evaluation harness that produces objective, auditable metrics on the four canonical clinical questions from the Merck Manual.
- Reuse of the `ClinicalResponse` schema and critic so safety is not an afterthought.

Without orchestration and measurement we cannot demonstrate end-to-end correctness, citation faithfulness, or safety intervention rate — all of which are required to justify scaling this pattern.

The four canonical clinical questions that serve as the stable evaluation benchmark are:
1. Protocol for managing sepsis in a critical care unit.
2. Common symptoms of appendicitis and whether it can be cured via medicine (and the definitive surgical procedure if not).
3. Effective treatments/solutions and possible causes for sudden patchy hair loss (alopecia areata).
4. Recommended treatments for a person who has sustained a physical injury to brain tissue resulting in temporary or permanent impairment of brain function (traumatic brain injury).

These questions (codified in `evaluation/fixtures.py`) provide a fixed, reproducible benchmark for the harness.

## Decision
Introduce a **LangGraph-orchestrated Clinical RAG Pipeline** that composes the existing retriever factory and safety layer into a single, stateful, critic-in-the-loop flow, together with a lightweight evaluation harness.

**Key elements of the decision:**
- New `generation/` module containing prompt templates (strict grounding + citation enforcement adapted from the medical notebook system message), a simple LCEL `create_generation_chain`, and a citation/disclaimer formatter.
- A thin `pipeline/graph.py` (orchestration layer only) that builds a LangGraph `StateGraph`. Nodes include at minimum: `retrieve`, `generate` (optional in early Phase 1), `apply_safety_layer` (reused), and `format_output`. The graph uses `ClinicalResponse` (ADR-006) as the canonical output schema.
- Phase 1 introduces a minimal LangGraph-compatible orchestration path that invokes the existing ADR-006 safety layer as a graph step. The Safety Critic is treated as a first-class explicit node with conditional review routing in Phase 2.
- Lightweight deterministic evaluation harness in `evaluation/` (and exposed via `run.py`): citation presence/recall, disclaimer compliance, `warning_level_summary` utilization, basic groundedness heuristic, end-to-end latency. Full RAGAS + LLM-as-judge metrics deferred.
- Factory / graph construction remains config-driven via `AppConfig` (strategy, safety profile, etc.) with optional overrides for tests.
- Strict dependency rule: `generation/` and `pipeline/graph.py` may import from `retrieval/`, `safety/`, and `core/`, but never the reverse. `safety/` stays independent. `pipeline/graph.py` contains only state, nodes, edges, and orchestration — it does not become a second factory (retrieval construction stays in `pipeline/factory.py`).
- Strict prompt and output contract: any generated clinical answer must pass through `apply_safety_layer` before formatting.

**Explicit non-goals for this ADR:**
- Full multi-agent CrewAI/AutoGen patterns or complex tool use (ADR-008+).
- Production-grade observability/tracing or Docker packaging (later infrastructure ADR).
- Replacing the deterministic critic with an LLM judge in Phase 1.

## Non-negotiable Invariants
- No generated clinical answer may be returned without `ClinicalResponse` validation.
- No generated clinical answer may bypass `apply_safety_layer`.
- Every final answer must include citations and a clinical-use disclaimer.
- Retrieval, safety, and evaluation components must remain independently testable.
- The evaluation harness must run against the four fixed Merck benchmark questions.

## Consequences

**Positive:**
- End-to-end auditable flow on real clinical questions with visible citations, disclaimers, and safety flags.
- Reuses every prior ADR artifact (config, hierarchical chunks + metadata, hybrid retriever, safety critic/schemas).
- LangGraph experience directly transferable to agentic/multi-agent work and senior job requirements.
- Evaluation harness provides objective evidence of quality — a portfolio differentiator.
- Maintains the "safety before generation" and "thin assembly layer" discipline. `pipeline/graph.py` stays orchestration-only.

**Negative / Trade-offs:**
- Adds orchestration complexity (LangGraph state machine) earlier than pure LCEL. We accept this because the critic-in-the-loop pattern is exactly what clinical safety demands and mirrors patterns already used in the delivery-exception multi-agent work.
- Generation remains intentionally limited in Phase 1; richer prompting / few-shot / tool use comes later. This is deliberate scope control.
- Evaluation starts deterministic and lightweight; more sophisticated metrics (RAGAS, clinical-specific faithfulness) are explicitly Phase 2 to avoid over-engineering the first working pipeline.

## Implementation Roadmap

**Phase 1 (this ADR — deliverable on this branch):**
- `generation/prompts.py` and `generation/chain.py` (strict system message + citation formatter + LCEL chain).
- `pipeline/graph.py` with minimal StateGraph that supports a retrieval + safety path; any generated answer must invoke `apply_safety_layer` before formatting. The Safety Critic becomes a first-class explicit node with conditional routing in Phase 2.
- `evaluation/harness.py` with the four core deterministic metrics + runner integrated into `run.py`.
- Updated demo that exercises the four Merck questions and prints structured `ClinicalResponse` output with safety flags.
- Unit tests for new modules and integration test that the full (safe) path produces grounded output.
- ADR-007 accepted and README updated.

**Phase 2 (next ADR or follow-up work):**
- Richer evaluation (citation faithfulness, disclaimer quality score, refusal rate on out-of-scope queries).
- Promote the graph to a full LangGraph pipeline with Safety Critic as an explicit node (conditional edges for human review on high-`warning_level_summary` cases).
- Configurable safety profiles ("strict clinical" vs "research mode").
- Optional human-in-the-loop review workflow stub.
- Observability hooks (LangSmith / Langfuse compatible tracing of safety interventions).

## Phase 1 Implementation Summary and Enhancements

During the implementation of this ADR, several refinements were made to the Generation Layer to improve clinical usefulness without altering the core LangGraph orchestration architecture.

### Structured Uncertainty Communication

The original binary approach ("answer confidently or refuse") was evolved into a more practical model:

- Extended `ClinicalResponse` with three new fields:
  - `uncertainty_note`
  - `key_findings_to_verify`
  - `recommended_next_steps`

- Updated `CLINICAL_SYSTEM_PROMPT` to instruct the model on three possible scenarios:
  1. Strong support in context → Answer directly with high confidence.
  2. Partial support → Provide the most likely interpretation, populate uncertainty fields, and set `requires_human_review: true`.
  3. No meaningful support → Return the standard out-of-knowledge refusal.

- Enhanced fallback logic in `generate_clinical_answer` to populate the new uncertainty fields when JSON parsing fails.

- Added schema validators to automatically escalate `requires_human_review` when uncertainty is detected or confidence is low.

These changes allow the system to deliver more clinically useful output while remaining transparent about limitations.

## Status

**Completed** — Phase 1 of ADR-007 is considered finished. The LangGraph orchestration layer, generation step, safety integration, and structured output (including uncertainty handling) are implemented and tested.

## Future Work (Phase 2)

The following items were originally envisioned as part of this ADR but are now deferred to a follow-up ADR (planned as ADR-008):

- Full evaluation harness (RAGAS metrics + clinical-specific evaluators such as citation faithfulness, warning grounding, and refusal quality).
- Potential introduction of a second specialized prompt for low-confidence scenarios.
- More advanced observability and tracing integration (beyond basic `@traceable` usage).
- Performance and retrieval quality improvements (embeddings, re-ranking, metadata boosting).

These items will be tracked in a new ADR to keep the scope of ADR-007 focused and complete.
