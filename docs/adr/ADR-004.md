# ADR-004: Metadata-Aware Hybrid Retrieval for Medical RAG

## Status
Draft (2026-06-03)

## Context
The ingestion pipeline (ADR-002 and ADR-003) produces high-quality chunks with rich metadata, including `section_hierarchy`, `chunk_type`, `warning_level`, and page information. The Merck Manual is a very large document (~2000+ pages) containing thousands of medical conditions with highly overlapping symptoms. This creates a challenging differential diagnosis scenario where pure vector similarity often fails to surface the most clinically relevant information.

A basic top-k vector search is insufficient for a medical RAG system that must support time-critical decisions by frontline healthcare workers. The system needs strong recall while maintaining precision, especially around high-risk information such as black-box warnings.

## Decision
We adopt **Metadata-Aware Hybrid Retrieval** as the primary retrieval strategy for the first production version.

### First Implementation Approach

- Use `langchain.retrievers.EnsembleRetriever` to combine:
  - Dense vector retrieval (FAISS + `BAAI/bge-m3`)
  - BM25 keyword retrieval
- Start with `k=8` candidates from the ensemble retriever
- Apply metadata-aware post-processing:
  - Boost scores for chunks where `chunk_type == "warning"`
  - Support optional metadata filtering (e.g., by section or `warning_level`)
- Return results with full metadata to enable high-quality, source-cited answers in the generation layer
- Implement this in a new `retrieval/` package following clean architecture principles

This approach leverages the rich metadata created during ingestion while adding the benefits of hybrid search without excessive complexity in the first version.

## Reasoning

Pure vector search struggles with medical terminology overlap and rare but critical information (such as warnings). BM25 complements dense retrieval by capturing exact keyword matches. Combining both through `EnsembleRetriever` provides a good balance between semantic understanding and lexical precision.

Metadata boosting (especially for warnings) ensures that high-risk clinical information is prioritized when relevant, which is essential for medical safety and trust in the system.

## Alternatives Considered

| Strategy                          | Pros                                      | Cons                                           | Decision |
|-----------------------------------|-------------------------------------------|------------------------------------------------|----------|
| Pure Vector Search (k=4~6)        | Simple, fast                              | Weak recall on overlapping medical symptoms    | Rejected |
| Vector Search + Metadata Filtering| Leverages existing metadata               | Still limited by vector-only matching          | Considered |
| **EnsembleRetriever (Vector + BM25)** | Good recall + precision, relatively simple | Requires tuning of weights                     | **Chosen for v1** |
| Full Custom Hybrid + Re-ranking   | Highest potential quality                 | Significantly more complex to implement well   | Future work |
| Pure BM25                         | Strong keyword matching                   | Weak semantic understanding                    | Rejected |

## Consequences

**Positive:**
- Significantly better retrieval quality than pure vector search for medical content
- Properly utilizes the rich metadata investment made in ADR-002
- Good balance between implementation effort and result quality
- Enables high-quality citations (section, page, warning status)
- Clear and maintainable architecture using LangChain components

**Negative / Trade-offs:**
- Adds some complexity compared to simple vector search
- Requires tuning of ensemble weights and score boosting factors
- `k=8` increases token usage (still acceptable within context limits)

**Future Work:**
- Dynamic `k` and ensemble weights based on query type
- More sophisticated re-ranking (e.g., cross-encoder or LLM-based)
- Section-aware or intent-aware retrieval strategies
- Evaluation harness using RAGAS or custom medical metrics
