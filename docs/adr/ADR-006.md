# ADR-006: Safety, Guardrails, and Output Validation Layer for Clinical RAG

**Status**: Accepted  
**Date**: 2026-06-10  
**Deciders**: Ivan Beira

## Context

Following ADR-005 (Lightweight RAG Pipeline Assembly), the project now has a clean, maintainable way to compose retrieval components. However, the system remains a **clinical decision-support tool** intended for medical professionals. 

In the medical domain, the cost of hallucination, poor source grounding, or missing critical warnings is significantly higher than in general-purpose RAG. At the same time, because this system is designed as a **helper for trained medical professionals** (not patients or the general public), overly restrictive guardrails that refuse high-risk clinical queries would reduce utility and trust.

ADR-005 already flagged the need for:
- Mandatory source citation with verifiable references
- Output guardrails / refusal mechanisms for unsafe or out-of-scope queries
- Comprehensive evaluation harness

This ADR formalizes the safety and validation layer that must sit between retrieval and final output generation. It must be implemented **before** any generation-oriented chains are considered production-ready.

## Decision

We will introduce a **dedicated Safety & Validation Layer** that operates after retrieval and before/around generation. The layer will be implemented primarily through:

- A new `safety/` package under `src/frontline_clinical_rag/`
- Structured Pydantic output schemas with validators
- A lightweight **Safety Critic** node (LangGraph-compatible)
- Mandatory disclaimer and citation enforcement

The philosophy is **"empower with guardrails, not block with refusals"** for legitimate clinical queries.

### Core Principles

1. **Do not refuse high-risk medical queries** — The system is intended for medical professionals. It should answer questions about sepsis protocols, trauma, drug interactions, etc., rather than deflect.
2. **Strong, non-negotiable disclaimers** — Every response must clearly state its limitations.
3. **Source grounding is mandatory** — No answer is complete without verifiable citations from the Merck Manual.
4. **Input protection only for malicious content** — Prompt injection and jailbreak detection remains, but clinical topics are never blocked.
5. **Post-generation validation over pre-generation refusal** — Use a Safety Critic to strengthen disclaimers, flag weak grounding, and suggest conservative language rather than refusing.

### Components of the Safety Layer

**1. Input Guardrail (Lightweight)**
- Detects prompt injection / jailbreak patterns only.
- Located in `safety/guardrails.py`.
- Early exit with safe logging if malicious input is detected.

**2. Structured Clinical Output Schema**
- New Pydantic model `ClinicalResponse` (or similar) in `safety/schemas.py`.
- Required fields:
  - `answer`
  - `sources` (list of `{page, section, excerpt}`)
  - `disclaimer` (standardized + context-aware)
  - `warning_level_summary`
  - `confidence` (self-reported by generator + validated)
  - `requires_human_review` (boolean flag)

**3. Safety Critic Node**
- A dedicated critic (similar to Project 3) that runs after generation.
- Responsibilities:
  - Verify source citations exist and are relevant.
  - Strengthen or add disclaimers when `warning_level` is high.
  - Detect potential over-confident language or missing caveats.
  - Suggest conservative phrasing ("consider", "per Merck Manual...", "always correlate clinically").
- Does **not** refuse the query — it improves the output.

**4. Output Formatter**
- Ensures every final response contains the standardized medical disclaimer at the top or bottom.
- Enforces citation format consistency.

### Dependency & Architectural Rules

- `safety/` package may depend on `core/config.py`, `retrieval/`, and future `pipeline/` components.
- `retrieval/` and `chunking/` **must not** import from `safety/`.
- The Safety Critic will be designed to work both inside a LangGraph pipeline and as a standalone post-processing step (for flexibility during early development).

### Clinical Safety & Production Readiness

This is a **clinical decision-support RAG system**, not a diagnostic tool.

> Any generation pipeline (future `create_clinical_rag_chain()`) **must** route through the Safety & Validation Layer. Responses that fail citation validation or contain high `warning_level` content must be clearly marked and include stronger disclaimers. The system must never present itself as a substitute for clinical judgment.

A clear, prominent disclaimer will be part of every generated response.

### Implementation Roadmap

**Phase 1 (this ADR)**:
- `safety/guardrails.py` (input injection scanner)
- `safety/schemas.py` (Pydantic `ClinicalResponse` model + validators)
- `safety/critic.py` (Safety Critic logic + prompt)
- Integration hook in `pipeline/factory.py` (optional `with_safety=True` path)
- Updated demonstration script that shows disclaimer + citations on the four clinical questions

**Phase 2 (next ADR)**:
- Full LangGraph pipeline that includes Safety Critic as a node
- Evaluation harness that measures citation faithfulness + disclaimer presence
- Configurable disclaimer profiles (strict vs. standard)

## Consequences

**Positive**
- Establishes responsible, auditable behavior from the start — critical for a medical portfolio project.
- Maintains high utility for medical professionals by avoiding unnecessary refusals.
- Creates clear extension points for future stricter modes (e.g., student vs. specialist).
- Improves output quality through the Safety Critic (better disclaimers, stronger grounding).
- Provides visible senior engineering thinking in the Git history.

**Negative / Trade-offs**
- Adds another layer that must be maintained and tested.
- The Safety Critic introduces some latency (acceptable for a clinical helper, but must be measured).
- Requires careful prompt engineering for the Critic so it improves rather than over-cautions every answer.
- Does not eliminate all risk — disclaimers are necessary but not sufficient.

## Alternatives Considered

| Approach                                      | Pros                                              | Cons                                                                 | Decision      |
|-----------------------------------------------|---------------------------------------------------|----------------------------------------------------------------------|---------------|
| Hard refusal on high-risk queries             | Strong safety signal                              | Reduces usefulness for the target audience (medical professionals)   | Rejected      |
| Only standardized disclaimer, no critic       | Simple and fast                                   | Weak grounding enforcement; no post-generation improvement           | Rejected      |
| Full agentic safety system (multiple critics) | Very robust                                       | Overkill for current scope; increases complexity early               | Rejected      |
| **Layered soft guardrails + Safety Critic**   | Balanced, useful, auditable, extensible           | Moderate added complexity                                            | **Chosen**    |
| Rely only on prompt engineering               | Zero extra code                                   | Not reliable or auditable; hallucinations still possible             | Rejected      |
