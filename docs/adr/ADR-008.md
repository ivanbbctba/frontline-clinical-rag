# ADR-008: Explicit Certainty-Aware Control Flow via Dedicated Assessment Node and Conditional Routing in Clinical RAG LangGraph

**Status**: Proposed  
**Date**: 2026-06-12  
**Deciders**: Ivan Beira  
**Related ADRs**: ADR-005 (Thin Factory Assembly), ADR-006 (Safety & Validation Layer), ADR-007 (Initial LangGraph Introduction)

---

## Context

The LangGraph implementation from ADR-007 was an intentionally early step. It successfully introduced structured generation, reused the ADR-006 safety layer, and ensured that generated answers passed through validation. However, several clinically critical behaviors still need to be elevated from defensive safeguards into explicit graph-level control flow:

- Certainty, confidence, and escalation decisions were driven primarily inside LLM prompts and the structured output parsing in `generate_clinical_answer`.
- The `ClinicalResponse` model contained defensive post-processing validators (`escalate_review_when_uncertain`) that correctly enforced safety invariants, but these validators cannot remain the primary mechanism for clinical routing.
- The graph had only one meaningful conditional (`generate_answer`). After `generate` → `apply_safety`, execution followed a linear path to `format_output`.
- As a result, important clinical policy decisions (low confidence, need for human review, insufficient context, out-of-scope queries) were non-deterministic, difficult to observe in LangSmith, and impossible to unit-test as routing logic.

In a safety-critical clinical decision-support system, control flow that affects safety posture must be explicit, versioned, traceable, and testable in code — not hidden in prompts or schema side effects.

This ADR focuses specifically on the **safety-routing layer**. Broader evaluation concerns (faithfulness scoring, clinical correctness, RAGAS-style metrics, LLM-as-judge) belong in a future evaluation ADR and are out of scope here.

---

## Decision

We perform a **complete, breaking refactor** of the clinical RAG LangGraph.

**New Architecture**:

- Introduce a dedicated **`assess_and_route`** node that acts as the single source of truth for clinical integrity assessment and routing.
- This node receives the `generated_response`, applies the safety layer, produces the canonical `safe_response`, builds a structured `assessment` object, and determines a `RoutingDecision`.
- All downstream behavior is driven by **explicit conditional edges** based on the assessment.
- Four specialized handler nodes make each clinical posture concrete:
  - `format_high_confidence`
  - `handle_low_confidence_escalation`
  - `handle_insufficient_context`
  - `handle_out_of_scope_refusal`
- `ClinicalRAGState` is enriched with `assessment`, `routing_decision`, and `routing_history` for full auditability and LangSmith tracing.
- The LLM remains responsible for content generation and self-reported confidence signals. Routing decisions are derived from the `assessment` object (which combines safety layer output and deterministic policy), not directly from prompt text.

This design breaks backward compatibility because the previous linear + prompt-centric approach was fundamentally misaligned with production safety requirements.

### Safety Assessment Contract

The `assessment` object produced inside `assess_and_route` is intentionally **lightweight and deterministic**. It contains only the fields required to make immediate routing decisions:

- `confidence`
- `requires_human_review`
- `has_uncertainty_signal`
- `source_count`
- `warning_level_summary`

More advanced evaluation fields (such as context sufficiency judgment, out-of-scope detection, or explanatory rationale) are intentionally excluded from this ADR and belong in the `evaluation/` package under a future ADR.

### Routing Precedence (Single Source of Truth)

Routing precedence is deterministic and implemented inside `_determine_routing_decision`:

1. Low confidence, uncertainty signals, or human-review flag → `LOW_CONFIDENCE_ESCALATION`
2. High-confidence cited answer → `HIGH_CONFIDENCE`

This order ensures that safety signals take priority in routing decisions.

### ClinicalResponse Validation Boundary

`ClinicalResponse` may retain defensive validation invariants (e.g., forcing `requires_human_review` on low confidence or uncertainty). These remain good safety engineering.

However, **graph-level routing must not depend on hidden schema mutation**. Confidence and uncertainty signals must be assessed explicitly inside `assess_and_route`.

### Compatibility & Non-Goals

- `run_clinical_rag_graph(...)` keeps a similar public signature.
- The `generate_answer=False` retrieval-only diagnostic path remains supported (it bypasses generation and assessment and is treated as diagnostic only).
- **Out of scope for this ADR**: multiple LLM critic agents, LLM-as-judge routing, human-in-the-loop UI, changes to retrieval/chunking/embedding strategy, and broad RAGAS-style evaluation metrics.

---

## Consequences

**Positive**
- Every clinically significant decision becomes a first-class, observable graph construct.
- LangSmith traces clearly show the exact routing decision taken.
- Routing logic becomes a pure, testable function.
- Clear separation: Generation owns content quality; Assessment + Graph owns clinical policy.
- Strong foundation for future safety orchestration patterns.

**Negative / Trade-offs**
- More nodes and conditional logic (mitigated by centralizing decisions in `assess_and_route`).
- Breaking change requires updates to callers and demos.
- Slightly higher implementation surface area (accepted to correct the original design flaw).

---

## Alternatives Considered

| Alternative                                      | Why Rejected |
|--------------------------------------------------|--------------|
| Keep prompt + Pydantic post-processing only      | Useful defensively, but insufficient as the final routing architecture |
| Add only one extra conditional after `apply_safety` | Too incremental; still left critical decisions implicit |
| Heavy parallel critic agents immediately         | Premature complexity; better addressed after deterministic routing is explicit |
| Single monolithic assessment node                | Violates separation of concerns and hurts testability |

---

## Implementation Notes

- New graph lives in `src/frontline_clinical_rag/pipeline/graph.py`.
- `RoutingDecision` is implemented as a `StrEnum`.
- All nodes and the routing function must be properly traceable with LangSmith.
- `_determine_routing_decision` is the single source of truth for routing precedence.

### Minimum Implementation Tests

- High `confidence` + no uncertainty signals → `HIGH_CONFIDENCE`
- Low `confidence` or `has_uncertainty_signal` = true → `LOW_CONFIDENCE_ESCALATION`
- Generated answers must pass through `assess_and_route` + safety layer before formatting
- `generate_answer=False` path returns retrieval results without creating a clinical answer
