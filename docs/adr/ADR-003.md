# ADR-003: Observability and Tunability for Medical Document Ingestion

## Status
Accepted (2026-06-03)

## Context
The HierarchicalMedicalChunker implemented in ADR-002 uses PyMuPDF layout analysis to build rich section hierarchy and metadata. While this approach delivers high-quality chunks, it introduces several points where processing can degrade silently:

- PDF cannot be opened by `fitz.open()`
- No usable font size or layout information is extracted from the document
- No headings are detected using font size and boldness heuristics

When ingesting large clinical references such as the Merck Manual (~2000+ pages), the lack of visibility into these failure modes makes debugging and quality assurance difficult. Additionally, behavioral parameters such as the heading length threshold and data directory were hardcoded, reducing flexibility for testing and future document types.

## Decision
We introduce **structured logging** and **externalized configuration** as first-class concerns in the ingestion pipeline:

1. **Structured Logging**
   - Add `logging.getLogger(__name__)` in `src/frontline_clinical_rag/ingestion/loader.py`
   - Emit `warning` level messages at key failure and fallback points inside `HierarchicalMedicalChunker._add_layout_section_metadata`:
     - Non-PDF or missing files
     - Failure to open PDF with PyMuPDF
     - No font size information extracted
     - No layout headings detected (falling back to text-based hierarchy)
   - Keep logging lightweight and non-intrusive (no JSON formatting or external dependencies at this stage).

2. **Externalized Configuration**
   - Move `raw_data_path` and `max_heading_length` from module-level constants into `Settings` (via `pydantic-settings` and `.env`).
   - Update `MedicalDocumentLoader.load_pdfs()` and heading detection functions to read from `settings`.

This decision treats observability and tunability as essential qualities for a medical-grade RAG ingestion system.

## Alternatives Considered

| Alternative                        | Pros                              | Cons                                      | Rejected because |
|------------------------------------|-----------------------------------|-------------------------------------------|------------------|
| Keep using `print()` statements    | Simple, no new imports            | Not suitable for testing or production    | Poor observability |
| Full structured logging (JSON + tracing) | Excellent for production     | Overkill at current stage                 | Premature complexity |
| Hardcode all parameters            | Fast to implement                 | Difficult to test different configurations | Reduces maintainability |
| **Structured logging + Settings**  | Good balance of visibility and simplicity | Slightly more code | **Chosen** |

## Consequences

**Positive:**
- Significantly improved debuggability when ingesting large medical documents (especially the Merck Manual).
- Easier experimentation with heading detection thresholds (`max_heading_length`).
- Clear separation between code and environment-specific configuration.
- Better support for future evaluation and quality assurance work.
- Strong portfolio signal: demonstrates production thinking around observability.

**Negative:**
- Introduces a small number of log statements that must be maintained.
- Requires updating `env.example` when new configurable parameters are added.

**Future work:**
- Consider adding log level configuration via environment variable.
- Evaluate structured JSON logging if we introduce evaluation pipelines or background jobs.
- Potentially add simple metrics (e.g., number of documents that fell back to text-based hierarchy).
